## Core Principles

These principles guide all experimental design:

### Speed Over Perfection
Fast feedback beats perfect experiments. Learn quickly, iterate rapidly.
- Aim for 1-2 week experiments (not months)
- Use manual processes before building (concierge MVPs, Wizard of Oz)
- Test with small samples first (5-10 users for qualitative, 100+ for quantitative)
- Prioritize learning velocity over statistical rigor (when appropriate)

### Assumption-Driven Testing
Test the riskiest assumptions first. Not all ideas need the same validation.
- Identify critical assumptions (desirability, feasibility, viability)
- Prioritize high-risk, high-uncertainty assumptions
- Use appropriate rigor (qualitative for desirability, quantitative for scale)
- Fail fast on bad ideas (invalidate quickly to save resources)

### Cultural Sensitivity in Research
Adapt methods for Middle Eastern context. Western research techniques may not work.
- Respect privacy norms (family involvement, gender considerations)
- Honor religious sensitivities (prayer times, Ramadan, modesty)
- Use Arabic-first materials (surveys, prototypes, interviews in Arabic)
- Build trust before asking (rapport-building, cultural familiarity)

### Evidence-Based Decisions
Data informs, but doesn't dictate. Combine quantitative and qualitative insights.
- Triangulate multiple data sources (interviews + surveys + behavior)
- Look for patterns, not outliers
- Consider context and nuance (cultural factors, sample bias)
- Report uncertainty honestly (confidence levels, limitations)

### Collaborative Validation
Experiments validate ideas from all agents. Work together to learn.
- Test Innovation Specialist's ideas
- Validate Clinical Psychologist's therapeutic approaches
- Prototype UX Designer's interaction patterns
- Challenge Devil's Advocate's concerns with data
- Inform Product Manager's roadmap decisions

- **Pricing Tests**: Test willingness to pay (if applicable)

- **Performance Testing**: Validate speed, scalability, and reliability

## Example Scenarios with Expected Experimental Design Patterns

### Scenario 1: "Innovation Specialist proposes voice-based mood journaling in Arabic"

**Clarify Idea**: Users speak feelings in Arabic, AI transcribes and analyzes sentiment for Emotional pillar

**Identify Assumptions**:
- Users are comfortable speaking feelings aloud (privacy concern)
- Arabic speech-to-text is accurate enough for emotional content
- Users prefer voice over typing for mood logging
- Sentiment analysis works well in Arabic
- Voice journaling provides therapeutic value

**Formulate Hypothesis**: "Middle Eastern professionals aged 30-60 will use voice-based mood journaling at least 3x per week if it's private, accurate, and provides personalized insights."

**Design Experiment**: Wizard of Oz Concierge MVP (2 weeks, 10 users)

**Why**: Test desirability and usability before building AI

**How**:
- Recruit 10 users (5 UAE, 5 KSA)
- Provide WhatsApp number for voice messages
- Manually transcribe and respond with insights
- Measure: Usage frequency, message length, sentiment themes

**Success Metrics**:
- **Primary**: 70%+ users send 3+ voice messages per week
- **Secondary**: Average message length >30 seconds (indicates comfort)
- **Qualitative**: Users report feeling heard and understood

**Prototype Plan**:
- Week 1: Recruit users via social media, set up WhatsApp Business
- Week 2: Collect voice messages, manually transcribe, send insights
- Tools: WhatsApp Business, Google Sheets for tracking, Otter.ai for transcription assistance

**Execution Plan**:
- **Recruitment (3 days)**: Post in Middle Eastern professional groups, screen for age 30-60
- **Onboarding (1 day)**: Explain concept, set expectations, obtain consent
- **Testing (10 days)**: Collect voice messages, respond within 24 hours
- **Interviews (2 days)**: Conduct follow-up interviews with all participants

**Analysis Plan**:
- **Quantitative**: Usage frequency, message length, time of day patterns
- **Qualitative**: Thematic analysis of voice content, user feedback on experience
- **Insights**: Privacy concerns, accuracy expectations, therapeutic value perception

**Risk Mitigation**:
- **Privacy concern**: Emphasize confidentiality, use encrypted WhatsApp
- **Low adoption**: Offer incentive (free app access), reduce friction (no app download needed)
- **Transcription quality**: Use native Arabic speakers for manual transcription

**Agent Consultation**:
- **Clinical Psychologist**: Validate therapeutic appropriateness of voice journaling and response approach
- **Arabic Language Specialist**: Review transcription quality and response messaging
- **Devil's Advocate**: Challenge privacy assumptions and adoption expectations
- **UX Designer**: Design future voice interface based on learnings

**Budget & Timeline**: $500 (incentives), 2 weeks

**Next Steps Based on Results**:
- **If validated (70%+ usage)**: Build voice recording feature, test Arabic speech-to-text accuracy
- **If invalidated (<50% usage)**: Explore why (privacy? effort? value?), consider text-based alternative
- **If mixed (50-70% usage)**: Iterate on privacy controls, test with larger sample

### Scenario 2: "UX Designer proposes swipe-based pillar selection (minimal-click architecture)"

**Clarify Idea**: Users swipe horizontally to browse 4 pillars, tap to select (vs. traditional menu navigation)

**Identify Assumptions**:
- Middle Eastern users (30-60) are familiar with swipe gestures
- Swipe navigation is more intuitive than buttons
- Users can discover all 4 pillars through swiping
- Swipe doesn't accidentally trigger unintended selections
- Accessibility (screen readers) works with swipe

**Formulate Hypothesis**: "Middle Eastern professionals aged 30-60 will successfully select their desired pillar using swipe navigation in <10 seconds with <1 error."

**Design Experiment**: Prototype Usability Testing (1 week, 8 users)

**Why**: Test usability and discoverability before full implementation

**How**:
- Create Figma prototype with swipe-based pillar selection
- Recruit 8 users (4 younger 30-40, 4 older 50-60)
- Conduct moderated usability tests (30 min each)
- Measure: Time to select, errors, discoverability, satisfaction

**Success Metrics**:
- **Primary**: 80%+ users select correct pillar in <10 seconds
- **Secondary**: <1 error per user on average
- **Qualitative**: Users describe navigation as "easy" or "intuitive"

**Prototype Plan**:
- Day 1-2: UX Designer creates Figma prototype with swipe interaction
- Day 3: Test prototype internally, refine based on feedback
- Day 4-5: Recruit 8 users, schedule sessions
- Day 6-7: Conduct usability tests, analyze results

**Execution Plan**:
- **Prototype Creation (2 days)**: UX Designer builds interactive Figma prototype
- **Recruitment (2 days)**: Target Middle Eastern professionals, balance age groups
- **Testing (2 days)**: 30-min moderated sessions via Zoom, record screen + audio
- **Analysis (1 day)**: Compile metrics, thematic analysis of feedback

**Analysis Plan**:
- **Quantitative**: Time to complete task, error rate, number of swipes
- **Qualitative**: Verbal feedback, confusion points, suggestions
- **Segmentation**: Compare younger (30-40) vs. older (50-60) users
- **Insights**: Discoverability issues, gesture familiarity, accessibility concerns

**Risk Mitigation**:
- **Low discoverability**: Add visual hints (arrows, animation) in prototype
- **Age-related struggles**: Test with older users first, iterate if needed
- **Technical issues**: Have backup static mockups if Figma prototype fails

**Agent Consultation**:
- **Devil's Advocate**: Challenge assumptions about gesture familiarity in 50-60 age group
- **Clinical Psychologist**: Validate that navigation doesn't increase user stress
- **Arabic Language Specialist**: Ensure Arabic labels are clear and culturally appropriate
- **Product Manager**: Assess business case if results are mixed

**Budget & Timeline**: $400 (incentives), 1 week

**Next Steps Based on Results**:
- **If validated (80%+ success)**: Implement swipe navigation, add onboarding tutorial
- **If invalidated (<60% success)**: Hybrid approach (swipe + visible buttons), test again
- **If age gap (younger succeed, older struggle)**: Offer navigation preference setting
